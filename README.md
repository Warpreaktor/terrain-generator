# terrain-generator

Библиотека и лаборатория для генерации рельефа в виде сетки высот.

Проект состоит из двух модулей:

- **terrain-core** — чистая библиотека работы с рельефом.
- **terrain-lab** — desktop-приложение на libGDX для визуализации, настройки и ручной правки рельефа.

Основная идея проекта: сначала построить качественный и универсальный инструмент генерации **формы поверхности**, а уже потом поверх него делать более высокоуровневые генераторы конкретных ландшафтов.

---

## 1. Текущее состояние проекта

На текущем этапе проект умеет:

- хранить рельеф как сетку высот;
- генерировать рельеф процедурно на основе параметров;
- использовать пресеты общего типа местности;
- визуализировать карту строго **в виде сверху**;
- показывать карту как **картографическую шкалу высот**;
- включать и выключать сетку, менять шаг сетки;
- редактировать рельеф вручную кистью поверх процедурной основы;
- работать с **нулевым уровнем высоты** как с опорной отметкой;
- отображать отрицательные и положительные высоты относительно этого нуля.

Проект **не** занимается:

- биомами;
- водой;
- минералами;
- породами;
- температурой;
- специфическими ландшафтами вроде "горячего источника";
- предметной логикой игрового мира.

Сейчас генератор отвечает только за **сам рельеф**.

---

## 2. Архитектура

### 2.1. Модуль `terrain-core`

`terrain-core` не должен зависеть от рендера и UI.

В модуле находятся:

- модель сетки высот;
- хранилище высот;
- операции над рельефом;
- шумовые функции;
- фильтры;
- анализ диапазона высот и статистики.

Основные сущности:

- `TerrainGrid` — сетка высот;
- `HeightStorage` / `DenseFloatHeightStorage` — хранение данных;
- `TerrainOperation` — одна операция над сеткой;
- `TerrainOperationPipeline` — последовательность операций;
- `TerrainStatistics` — статистика рельефа.

Основной принцип:

> `terrain-core` знает только геометрию поверхности и операции над ней.

---

### 2.2. Модуль `terrain-lab`

`terrain-lab` — это лаборатория и визуальный стенд.

Он нужен для того, чтобы:

- видеть результат генерации глазами;
- крутить параметры;
- переключать пресеты;
- тестировать кисти ручной правки;
- быстро находить архитектурные и математические ошибки.

`terrain-lab` зависит от `terrain-core`, но не наоборот.

Основные задачи модуля:

- top-down отрисовка рельефа;
- UI для параметров генерации;
- toolbar и вспомогательные окна;
- статусная панель;
- ручное редактирование рельефа.

---

## 3. Модель высоты

### 3.1. Signed-высота

В проекте принята **знаковая модель высоты**.

Это значит:

- `0` — опорный уровень;
- отрицательные значения — ниже опорного уровня;
- положительные значения — выше опорного уровня.

Это сделано специально, чтобы любой внешний клиент библиотеки мог трактовать `0` по-своему:

- как уровень моря;
- как базовый уровень суши;
- как нулевую отметку мира;
- как условную среднюю поверхность;
- как любую другую модельную границу.

### 3.2. Отображение высоты

Для визуализации рельефа используется не физический смысл цвета, а нормализованное представление относительно диапазона отображения.

Это значит:

- тёмные области показывают более низкие высоты;
- светлые области показывают более высокие высоты;
- `0` не обязан быть ни белым, ни чёрным;
- визуальный масштаб должен оставаться стабильным, чтобы локальная правка кистью не "ломала" восприятие всей карты.

## 4. Процедурная генерация

Сейчас процедурный слой не должен генерировать один конкретный тип формы. Его задача — уметь выдавать разные морфологии местности.

Генератор опирается на набор параметров, а не на один жёстко прошитый сценарий.

### 4.1. Что должно получаться

На основе параметров и пресетов генератор должен уметь выходить на:

- почти ровную поверхность;
- слабоволнистую равнину;
- холмистую местность;
- грядовую местность;
- сильно пересечённый рельеф;
- гористую местность.

### 4.2. Пресеты

Пресеты — это не отдельные алгоритмы, а наборы параметров общего генератора.

На текущем этапе используются такие типы пресетов:

- `plain`
- `rolling plains`
- `hills`
- `ridge field`
- `mountains`
- `custom`

`custom` означает, что пользователь ушёл от заводских значений и работает с параметрами вручную.

### 4.3. Параметры генерации

Сейчас лаборатория оперирует следующими параметрами:

- `relief amplitude` — общий масштаб перепадов высот;
- `macro scale` — масштаб крупных форм;
- `detail scale` — масштаб мелких деталей;
- `roughness` — шероховатость рельефа;
- `smoothness` — сила сглаживания;
- `slope angle` — направление уклона;
- `slope strength` — сила глобального уклона;
- `ridge strength` — влияние грядовой составляющей;
- `elevation bias` — смещение распределения высот.

Этот набор считается рабочим стартовым набором, но не окончательным.

Возможные будущие расширения:

- `valley bias`
- `plateau bias`
- `anisotropy`
- `directional stretch`
- `fragmentation`
- `warp strength`
- `warp scale`

## 5. Ручная правка рельефа

Помимо процедурной генерации в проекте есть второй слой работы с картой — **ручная правка**.

Это отдельный слой поверх процедурной основы.

### 5.1. Зачем он нужен

Ручная правка нужна для двух задач:

1. быстро доводить рельеф локально;
2. нащупывать, каких операций не хватает процедурному генератору.

### 5.2. Текущие инструменты

Сейчас используются кисти:

- `raise`
- `lower`
- `smooth`
- `flatten`

Параметры кисти:

- радиус;
- сила;
- целевая высота для `flatten`.

### 5.3. Принцип работы

Ручная правка не должна ломать весь мир из-за одного локального изменения.

Поэтому:

- кисть действует локально;
- визуальный диапазон карты должен оставаться стабильным;
- слой ручной правки должен корректно работать со signed-высотой.

## 6. UI и взаимодействие

### 6.1. Общий подход

Интерфейс строится по редакторному принципу:

- сверху располагается **панель инструментов**;
- дополнительные окна вызываются по необходимости;
- карта остаётся основной рабочей областью.

Это сделано по аналогии с графическими редакторами: сначала выбор режима и инструмента, затем локальная работа на холсте.

### 6.2. Верхняя панель инструментов

На верхней панели доступны:

- **Generator** — открыть окно параметров генерации;
- **Sculpt** — открыть окно параметров ручной правки;
- **Status** — показать или скрыть статусную панель.

### 6.3. Окна поверх карты

Сейчас в `terrain-lab` используются overlay-окна:

- окно генератора;
- окно sculpt-настроек;
- статусная панель.

Они не должны блокировать всю карту только потому, что открыты. Работа по карте должна блокироваться только если указатель мыши находится непосредственно над UI-элементом.

---

## 7. Статусная панель

Статусная панель показывает текущую техническую информацию о сессии.

Ожидаемый состав данных:

- seed;
- режим работы;
- активный пресет;
- состояние auto-apply;
- текущая палитра;
- режим и шаг сетки;
- режим просмотра;
- диапазон высот;
- координаты курсора по сетке;
- высота клетки под курсором.

Статусная панель должна уметь скрываться, чтобы не перекрывать карту.

---

## 8. Управление в лаборатории

Ниже собраны актуальные договорённости по управлению. Если код в конкретной сборке временно отстаёт от этого описания, ориентиром всё равно считается именно эта модель поведения.

### 8.1. Общие действия

- `F1` — показать или скрыть окно управления;
- `F2` — показать или скрыть статусную панель;
- `R` — сгенерировать новый мир с новым seed;
- `W`, `A`, `S`, `D` — перемещение камеры;
- `Q`, `E` — zoom.

### 8.2. Отображение

- `C` — переключение палитры;
- `G` — переключение режима сетки;
- `H` — переключение шага сетки.

### 8.3. Режимы работы

- `Generator` — работа с параметрами процедурного генератора;
- `Sculpt` — работа кистью по карте.

### 8.4. Работа кистью

Типичное поведение режима sculpt:

- ЛКМ — применить кисть;
- инструмент выбирается в sculpt-окне;
- радиус и сила задаются через UI;
- `flatten target` задаёт уровень выравнивания.

## 9. Сборка и запуск

### 9.1. Требования

- Java 21
- Gradle 8.x

### 9.2. Сборка

В корне проекта:

```bash
./gradlew build
```

### 9.3. Запуск лаборатории

```bash
./gradlew :terrain-lab:run
```

### 9.4. macOS

Для macOS при запуске LWJGL3 нужен параметр:

```text
-XstartOnFirstThread
```

Этот параметр должен быть добавлен в задачу запуска `terrain-lab`, иначе приложение может не стартовать корректно.

## 11. Принципы разработки

При развитии проекта важно соблюдать следующие правила.

### 11.1. Никакой предметной логики в `terrain-core`

В библиотеку не должны просачиваться:

- горячие источники;
- минералы;
- болота;
- геотермия;
- вода;
- биомы;
- игровые объекты.

`terrain-core` работает только с формой поверхности.

### 11.2. Не хардкодить бессмысленные числа

Если в формуле используется число, оно должно:

- либо приходить параметром;
- либо быть вынесено в локальную переменную с понятным именем;
- либо быть оформлено как константа, если используется повторно.

Математика не должна быть записана как набор "магических" литералов.

### 11.3. Поддерживать signed-модель высоты

Нельзя возвращаться к модели, где вся высота живёт только в `0..1`.

`0` — опорный уровень.

Это принципиальная часть архитектуры.

### 11.4. Лаборатория не должна диктовать библиотеке архитектуру

UI, панели и hotkeys нужны для удобства тестирования, но они не должны загрязнять `terrain-core` зависимостями от libGDX, Scene2D и конкретных виджетов.

## 12. Известные риски и зоны доработки

Проект находится в активной фазе сборки архитектуры, поэтому некоторые части ещё будут дорабатываться.

На ближайшем горизонте:

- стабилизировать поведение UI-кнопок и окон;
- довести логику auto-apply / manual apply;
- дополнительно отполировать режим sculpt;
- расширить морфологические параметры генератора;
- подумать над сохранением и загрузкой конфигураций;
- подготовить экспорт карты и параметров генерации.

## 13. Ближайшая цель проекта

Ближайшая практическая цель:

> Получить универсальную библиотеку генерации рельефа, которая умеет строить качественную heightmap-поверхность, управляется параметрами, поддерживает локальную ручную правку и может служить основой для более высокого генератора игровых ландшафтов.

И только после этого имеет смысл поднимать следующий слой:

- воду;
- климат;
- геотермию;
- химические поля;
- биомы;
- живые сущности.

Сначала поверхность. Потом всё остальное.
